About protocols.js

DocumentationSVN

This library was designed to support complex and large entreprise web 2.0 applications.

It is quick to learn, fast to run and safe to depend on.

Web Protocols

Like most AJAX libraries protocols.js attempts to bridge the compatibility gap between browsers, although on a smaller set of interfaces, providing conveniences to leverage a stable stack of web protocols: HTTP, HTML, CSS and JSON.

Call it SOA, ROA or - more appropriately - a Representational State Transfer architecture, the design of a web applications is constrained first by the protocols they implement. So, your mileage of application server systems, languages or APIs may vary, if their protocols do not.

For JavaScript

The 700th implementation of a Lisp interpreter in C clothing does not suck.

So, I did not bend JavaScript like Ruby (prototype), Python (mochikit) or Java (Dojo).

Nor did I try to reimplement yet another framework for throwable web applications. If you need one, use jQuery. This library was writen to get entreprise web applications developers where less is more, now and in long run.

The absence of a ton of abstractions or another language emulation makes protocol.js easier to learn and ensure that its applications are reliable, load fast and run quick enough.

Applications

I use protocols.js to develop model-driven user interfaces for web application services programmed in JavaScript, Java or CPython.

Positionning, effects, drag & drop or most browser bugs are not supported: entreprise applications should not be developped on broken programming interfaces or depend all the HTML and CSS incompatibilies between web browsers.

Making an application look good in the eyes of its users demands to make it perform quickly and without fault. Dressing up ad-hoc HTML templates and sleek CSS styles on every browsers supported can be done later, concurrently and without changes to JavaScript programs.
