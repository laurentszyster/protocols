If the network is the computer then protocols are its application programming interfaces.

Protocols lets architects, programmers and designers leverage a stable stack of protocols to produce together large and complex applications made of many simple parts.

Web Protocols

Like most AJAX libraries protocols.js attempts to bridge the compatibility gap between browsers, although on a smaller set of interfaces, providing conveniences to leverage four defacto standard web protocols: HTTP, HTML, CSS and JSON.

Call it SOA, ESB or - more appropriately - a Representational State Transfer architecture, the design of a web applications is constrained first by the protocols they implement. Your mileage of application server systems, languages or APIs may vary, if their protocols do not.

To deliver an application that meets its entreprise requirements on time and budget, protocols.js supports a model driven architecture where blueprints are written directly using the text protocols applied: JSON for object modeling; CSS and HTML for presentation; HTTP for network integration.

For JavaScript

The 700th implementation of a Lisp interpreter in C clothing does not suck.

So, I did not bend JavaScript like Ruby / Prototype, Python / Mochikit or Java / Dojo.

Nor did I try to reimplement yet another framework for throwable web applications. If you need one, use jQuery, Ext and YUI. This library was made to get entreprise web applications developers where less is more, now and in long run.

Visual effects are tricky to run reliably accross browsers, drag and drop has allways been a gimmick with too many pitfalls for its users and GUI widgets in a web browser are pure non-senses.

This library won't raise back the client/server zombie APIs back from the 80's.

The absence of a ton of abstractions, a questionable widgetry or another language emulation makes protocol.js easier to learn and ensures that its applications are reliable, small and fast enough.
